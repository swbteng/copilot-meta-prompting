### 10. VS Code 확장 P3(셋업 마법사) 구축
- **수동 셋업(README 빠른 시작 2~3번)을 확장이 대신 실행**. 새 파일 `src/setup.js`(SetupWizard):

  `proxyStrictSSL` 설정->재시작 모달, 완료를 `globalState`에 저장), `cleanup()`(인증서 `-delstore`+

- **인증서 확보**: `~/.mitmproxy/mitmproxy-ca-cert.{pem,cer}` 사용(pem=NODE_EXTRA_CA_CERTS 필수, cer=certutil).
  없으면 엔진을 잠깐 띄워 생성하고 최대 12초 폴링. CN=mitmproxy(되돌리기 `-delstore Root mitmproxy`).
- **재시작 불가피**: `NODE_EXTRA_CA_CERTS`는 프로세스 시작 시 1회만 읽힘 -> 창 새로고침 아닌 **완전 종료
  후 재시작**을 안내(모달의 "지금 VS Code 종료"=`workbench.action.quit`). 현재적용 여부는
  `process.env.NODE_EXTRA_CA_CERTS`로, 영구설정은 `reg query HKCU\Environment`로 판별.
- **프록시 잔존(먹통) 방지**: 사용자가 엔진을 끄면(토글/끄기 명령) `http.proxy`가 127.0.0.1을 가리킬 때
  "프록시 해제?"를 물어봄(extension.js의 userStopEngine/userToggleEngine). dispose() 종료 경로는 경고 안 함.
- 안전: 모든 외부 명령은 `_exec()`로 감싸 **throw 금지**(fail-open). `certutil`/`reg` 실패는 보조라 계속,
  `setx` 실패만 셋업 중단. 정제 로직(refine_copilot.py)은 일절 미수정 - OS/에디터 설정만 변경.
- `package.json`: 명령 3개(setup/setupStatus/cleanupSetup) 추가, 제목줄 톱니(⚙=setup) navigation@4,
  버전 0.2.0 -> 0.3.0. P2 누락분: `logDir` 설명을 실제 동작(globalStorage 단일)으로 정정.

  `--force`, 남에게 '업데이트'로 보이려면 `package.json version` 올리기. 현재 vsix는 엔진 미동봉(P5에서 동봉).

### 13. VS Code 확장 P5(엔진 동봉 -> 자기완결 vsix) 구축
- **목표 달성**: "확장만 설치" - 받는 사람이 파이썬/저장소 없이 **vsix 설치 -> 셋업 -> 재시작**이면 끝.
- **standalone mitmdump 동봉**: `downloads.mitmproxy.org/12.2.3/mitmproxy-12.2.3-windows-x86_64.zip`(85MB)
  에서 `mitmdump.exe`(27.5MB)만 추출 -> `extension/engine/`. **자기완결 확인**: `mitmdump.exe --version`이
  `Mitmproxy: 12.2.3 binary / Python: 3.14.4`(venv의 3.13.13이 아닌 **내장 파이썬**) 출력 = Python 불필요.
- **애드온 동봉**: 루트 `refine_copilot.py` -> `engine/refine_copilot.py`(build.sh가 매 빌드 복사, 항상 동기화).
- **런타임 연결 기존 그대로**: config.resolveMitmdump/resolveAddon가 이미 `extensionUri/engine/`를 1순위로 봄
  -> 코드 수정 없이 설치본이 동봉 엔진을 찾는다(워크스페이스 안 열어도 됨).
- **build.sh 확장**: `ensure_engine`(애드온 복사 + mitmdump 없으면 다운로드, 캐시) + `--refresh-engine` 플래그.
  `.vscodeignore`에 `engine/**/*.zip` 안전장치(임시 zip 제외), `engine/mitmdump.exe`·`refine_copilot.py`는 포함.
- `package.json` 0.4.0 -> 0.5.0.
- **검증**: 동봉 바이너리 standalone 실행(--version) OK + **스모크 `_selftest_engine_bundle.js`**(동봉
  `mitmdump.exe -s engine/refine_copilot.py --listen-port 8097`): 애드온 로드 + 포트 바인딩 + 3s 생존 =
  **PASS**(검증 후 `__pycache__`/`logs`/셀프테스트 정리). `./build.sh` 재빌드 결과 **vsix에 engine/mitmdump.exe
  (26.24MB) + engine/refine_copilot.py 포함, 총 ~25.7MB** 확인.
- **주의(공유 전)**: `refine_text()` 고정문자열 -> 정식 서버 전환은 여전히 사용자 지시 후. `engine/`은 빌드
  산출물(gitignore) - 애드온은 루트본만 수정(AGENTS.md §5에 명시).

### 14. P6 계획 확정 (`@refine` Chat Participant, 확인형 정제) — 문서화만, 미구현
- 사용자 스펙 확정: VS Code 우측 Copilot Chat에서 `@refine <프롬프트>` -> 기존 refiner처럼 정제 ->
  **정제본 미리보기 + 5버튼**으로 승인 게이트 -> 클릭대로 동작. 기존 켜짐/꺼짐(자동) 모드와 **공존**.
- **5버튼**: allow(정제본 전송, 프록시 bypass) / try again(**정제 안 된 원본**으로 재정제 후 재확인) /
  use original prompt(구 "use my prompt", 원본 전송·정제 없음·bypass) / **modify**(정제본을 입력창에 내려
  유저가 약간 수정 -> 엔터 시 allow와 동일·bypass) / cancel(원본만 입력창에 시드 -> 엔터 시 **처음 @refine
  호출과 동일 취급**=재진입).
- **핵심 설계 = 프록시 bypass**: @refine가 이미 정제했으니 자동 프록시가 또 정제하면 안 됨 ->
  **refine_copilot.py에 bypass 마커(sentinel) allowlist 규칙 추가**(마커 있으면 제거 후 통과, fail-open 유지,
  messages/input·WS 양쪽). 이 덕에 **자동 Refiner 켜둔 채로도 @refine 공존**.
- **슬래시 컨벤션 확정(사용자 협의)**: 기본 진입=슬래시 없는 `@refine <프롬프트>`(VS Code 관용), 정제
  건너뛰기 전송은 **`/bypass` 하나로 통일**. 사용자 제안 `@refine /send` 기본안 대신 이 관용 컨벤션 채택
  ("더 강한 컨벤션 따르기" 지시). 구현 메모: allow/use-original=`@refine /bypass <확정텍스트>` 프로그램적
  제출(마커 확실), modify=입력창에 `@refine /bypass <정제본>` 시드(편집 후 엔터 시 같은 핸들러가 마커 부착),
  cancel=`@refine <원본>` 시드(슬래시 없음 -> 기본 흐름 재진입, 마커 불필요).
- 아키텍처: Chat Participant API(`createChatParticipant`/`stream.button`/`workbench.action.chat.open`,
  정식 API만), 정제는 JS에서(서버 HTTP 직접 호출/고정문자열), `engines.vscode ^1.90+`, 전송은 네이티브
  코파일럿 위임(평소 답 유지). 노력 중간 + refine_copilot.py 소규모 변경 동반(동봉본 build.sh 반영 확인).
- 상세: `docs/VSCODE_EXTENSION_PLAN.md`의 "P6 (계획, 미구현)" 섹션. **구현은 사용자 지시 후.**
